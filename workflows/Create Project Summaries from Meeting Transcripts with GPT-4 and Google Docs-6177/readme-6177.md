# Create Project Summaries from Meeting Transcripts with GPT-4 and Google Docs

---

### 1. Workflow Overview

This workflow automates the creation of a structured project summary document from a meeting transcript stored in Google Docs. Its primary use case is to streamline post-meeting documentation by extracting key project insights from transcripts and generating professional summaries using GPT-4, then saving the result as a new Google Docs file.

The workflow is logically divided into these blocks:

- **1.1 Input Reception:** Manual trigger to start the process and fetch the meeting transcript from Google Docs.
- **1.2 AI Processing:** Use GPT-4 to generate a detailed project summary based on the transcript.
- **1.3 Content Conversion:** Convert the AI-generated Markdown summary into HTML format.
- **1.4 Document Preparation:** Prepare a multipart HTTP request for creating a new Google Docs document with styled content.
- **1.5 Google Docs Creation:** Upload the formatted summary to Google Drive as a Google Docs file.
- **1.6 Finalization:** End the workflow with a no-operation node for graceful completion.

- **Nodes Involved:**  
  - When clicking ‘Execute workflow’  
  - Get meeting transcript

- **Node Details:**

    - Type: Manual Trigger

    - Inputs: None  
    - Outputs: Triggers the next node  
    - Edge Cases: None typical; user must manually trigger for execution

  - **Get meeting transcript**

    - Role: Retrieves the content of a Google Docs file containing the meeting transcript  
    - Config: Operation set to “get”, with a hardcoded document URL referencing the transcript file

    - Credentials: Google Docs OAuth2 (specific account configured)  
    - Edge Cases:  
      - Access denied or expired OAuth credentials  
      - Document not found or invalid URL  
      - Large documents causing timeouts or truncated content

#### 1.2 AI Processing

- **Overview:** Processes the meeting transcript with GPT-4 to generate a structured project summary in Markdown format, following a detailed prompt template.
- **Nodes Involved:**  
  - Project Summary  
  - OpenAI Chat Model

- **Node Details:**

  - **OpenAI Chat Model**  
    - Type: Langchain OpenAI Chat Model node  
    - Role: Executes GPT-4 (version “gpt-4.1-mini”) to process prompts and generate text  
    - Config:  
      - Model: gpt-4.1-mini  
      - Timeout: 5 minutes (300,000 ms)  
      - Max tokens: 32,768 (very high to accommodate long summaries)  
      - Max retries: 2  
    - Inputs: Triggered externally or from previous nodes (not explicitly connected in this workflow, but configured)  
    - Outputs: AI-generated text passed to the next node  
    - Credentials: OpenAI API key configured  
    - Edge Cases:  
      - API rate limits or quota exceeded  
      - Network timeouts  
      - Prompt formatting errors causing incoherent output

  - **Project Summary**  
    - Type: Langchain Chain LLM node  
    - Role: Defines the prompt and sends the transcript data to GPT-4 to produce a structured project summary in Markdown  
    - Config:  
      - Prompt instructs GPT-4 to act as a professional project manager/planner  
      - Provides example structure including headings: Project Overview, Key Challenges, Deliverables, Business Impact, Timeline  
      - Injects today’s date dynamically (`{{ $today }}`)  
      - Passes transcript content via `{{ $json.content }}`  
      - Output is Markdown text without code block delimiters  
    - Inputs: Receives transcript JSON from “Get meeting transcript”  
    - Outputs: Markdown text summary  
    - Edge Cases:  
      - Transcript missing or empty content  
      - Prompt template misalignment causing incomplete summaries

#### 1.3 Content Conversion

- **Overview:** Converts the Markdown summary generated by the AI into HTML with emoji and table support for styling in Google Docs.
- **Nodes Involved:**  
  - Markdown

- **Node Details:**

  - **Markdown**  
    - Type: Markdown converter  
    - Role: Converts Markdown text to HTML with formatting options  
    - Config:  
      - Mode: markdownToHtml  
      - Options: emoji and tables enabled  
      - Source Markdown: from `$json.text` (output of Project Summary)  
      - Destination key: `html_content`  
    - Inputs: Markdown summary from “Project Summary”  
    - Outputs: HTML formatted content  
    - Edge Cases:  
      - Invalid Markdown syntax causing conversion issues  
      - Unsupported Markdown extensions or embedded elements

#### 1.4 Document Preparation

- **Overview:** Constructs a multipart HTTP request body with metadata and styled HTML content, formatted for Google Drive API to create a new Google Docs file.
- **Nodes Involved:**  
  - set_fields  
  - Prepare_Request

- **Node Details:**

  - **set_fields**  
    - Type: Set node  
    - Role: Sets key metadata fields for document creation, including document name and target Google Drive folder ID  
    - Config:  
      - `document_name` assigned as the string “Project Summary”  
      - `google_drive_folder_id` assigned dynamically as `driveFolderId` (likely from external data or environment)  
    - Inputs: HTML content from “Markdown”  
    - Outputs: Metadata fields forwarded to next node  
    - Edge Cases:  
      - Missing or invalid folder ID causing upload failures

  - **Prepare_Request**  
    - Type: Code node (JavaScript)  
    - Role: Builds the multipart/related HTTP request body with boundaries, including JSON metadata and HTML document content with inline CSS for spacing and styles  
    - Configuration highlights:  
      - Boundary defined as “divider”  
      - Document metadata JSON includes `name`, `mimeType` (Google Docs type), and `parents` (folder ID)  
      - HTML is wrapped in a full document structure with `<style>` to add margins and spacing for paragraphs, headers, and lists  
      - Constructs multipart body with CRLF line breaks as required by Google Drive API  
      - Returns `rawData` property containing the full request body string  
    - Inputs: Metadata and HTML content from “set_fields” and “Markdown”  
    - Outputs: Raw HTTP request body for upload  
    - Edge Cases:  
      - Syntax errors in JavaScript code  
      - Missing or malformed HTML content causing upload to fail  
      - Incorrect boundary formatting causing API errors

#### 1.5 Google Docs Creation

- **Overview:** Uploads the prepared multipart request to Google Drive API to create a new Google Docs document containing the summarized project content.
- **Nodes Involved:**  
  - CreateGoogleDoc

- **Node Details:**

  - **CreateGoogleDoc**  
    - Type: HTTP Request node  
    - Role: Sends POST request to Google Drive API endpoint for file upload  
    - Config:  
      - URL: `https://www.googleapis.com/upload/drive/v3/files`  
      - Method: POST  
      - Query parameters: `uploadType=multipart`, `supportsAllDrives=true`  
      - Body: Raw, content-type `multipart/related; boundary=divider`  
      - Authentication: Google Drive OAuth2 credential configured for the user’s account  
      - Sends the `rawData` from “Prepare_Request” as the request body  
    - Inputs: Raw request body from “Prepare_Request”  
    - Outputs: Response from Google Drive API (file metadata)  
    - Edge Cases:  
      - OAuth token expiration or invalid scopes  
      - API quota limits  
      - Network failures or malformed request errors

#### 1.6 Finalization

- **Overview:** Marks the successful end of the workflow with a no-operation node.
- **Nodes Involved:**  
  - No Operation, do nothing

- **Node Details:**

  - **No Operation, do nothing**  
    - Type: NoOp node  
    - Role: Placeholder to end the flow gracefully after document creation  
    - Config: No parameters  
    - Inputs: From “CreateGoogleDoc” node  
    - Outputs: None  
    - Edge Cases: None
