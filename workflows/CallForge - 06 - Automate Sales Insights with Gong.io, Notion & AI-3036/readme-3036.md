# CallForge - 06 - Automate Sales Insights with Gong.io, Notion & AI

### 1. Workflow Overview

This workflow, **CallForge - AI-Powered Sales Call Data Processor**, automates the extraction, categorization, and storage of sales call insights generated by AI transcription and analysis tools such as Gong.io, Fireflies.ai, or Otter.ai. It targets sales teams, sales operations managers, and revenue teams who want to streamline the processing of sales call intelligence by automatically identifying competitor mentions, objections, integrations, and customer use cases, then storing these structured insights in Notion databases.

The workflow is logically divided into the following blocks:

- **1.1 Input Reception and Trigger**: Receives AI-generated sales call data from an external source.
- **1.2 Core AI Data Update**: Updates the main Notion call record with summarized AI insights.
- **1.3 Competitor Data Processing**: Checks for competitor mentions, splits and processes each competitor entry, and creates corresponding Notion database pages.
- **1.4 Integration Data Processing**: Checks for integration mentions, splits and processes each integration entry, and creates corresponding Notion database pages.
- **1.5 Objection Data Processing**: Checks for objections, splits objection tags, formats them, updates Notion with objection tags, and updates the call record with objection summaries.
- **1.6 Use Case Data Processing**: Checks for customer use cases, splits and processes each use case, and creates corresponding Notion database pages.
- **1.7 Rate Limiting and Sequencing**: Implements wait nodes to handle API rate limits and ensure smooth sequencing of Notion API calls.

### 2. Block-by-Block Analysis

#### 2.1 Input Reception and Trigger

- **Overview:**  
  This block receives AI-generated sales call data from an external workflow or webhook trigger, serving as the entry point for the entire process.

- **Nodes Involved:**  
  - Execute Workflow Trigger

- **Node Details:**  
  - **Execute Workflow Trigger**  
    - Type: `executeWorkflowTrigger`  
    - Role: Entry point node that triggers the workflow when AI call analysis data is received.  
    - Configuration: No parameters; expects incoming JSON with AI output and Notion metadata.  
    - Inputs: External trigger (webhook or workflow call).  
    - Outputs: Passes AI data to downstream nodes for processing.  
    - Edge Cases: Missing or malformed AI data could cause downstream failures.  
    - Sub-workflow: This node is the entry point and may be invoked by other workflows.

---

#### 2.2 Core AI Data Update

- **Overview:**  
  Updates the main Notion call record with summarized AI insights such as call summary, next steps, sentiment, and customer pain points.

- **Nodes Involved:**  
  - Update Notion Call object with AI data

- **Node Details:**  
  - **Update Notion Call object with AI data**  
    - Type: Notion (databasePage update)  
    - Role: Updates the existing Notion page representing the sales call with AI-generated summary fields.  
    - Configuration:  
      - Page ID dynamically set from trigger node's Notion data.  
      - Updates properties: Call Summary, Next Steps (formatted as bullet points), Sentiment (select), Customer Pain Points (formatted as bullet points).  
    - Inputs: AI output JSON from trigger node.  
    - Outputs: Passes data downstream for further processing.  
    - Edge Cases: Notion API errors, invalid page ID, or missing AI fields.  
    - Credentials: Uses Notion API credentials named "Notion david-internal".  
    - Rate limiting handled by wait nodes downstream.

#### 2.3 Competitor Data Processing

- **Overview:**  
  Checks if competitor data exists in AI output, then splits each competitor entry, creates Notion pages for each, and aggregates data.

- **Nodes Involved:**  
  - Check if Competitor Data Found  
  - Wait for rate limiting - Competitors  
  - Get AI Response - Competitors  
  - Split Out Competitor Data  
  - Create Competitors in DB  
  - Bundle Competitor Data to 1 object  
  - Merge Competitor Thread

- **Node Details:**  
  - **Check if Competitor Data Found**  
    - Type: If node  
    - Role: Checks if the AI output contains one or more competitor entries.  
    - Condition: Length of `AIoutput.Competitors` array ‚â• 1.  
    - Outputs: True branch triggers competitor processing; false branch skips.  
  - **Wait for rate limiting - Competitors**  
    - Type: Wait node  
    - Role: Pauses execution for 3 seconds to avoid Notion API rate limits.  
  - **Get AI Response - Competitors**  
    - Type: Set node  
    - Role: Extracts AI output for competitors into a variable for downstream use.  
  - **Split Out Competitor Data**  
    - Type: SplitOut node  
    - Role: Splits the competitor array into individual items for processing.  
  - **Create Competitors in DB**  
    - Type: Notion (databasePage create)  
    - Role: Creates a new page in the Competitors database for each competitor.  
    - Configuration:  
      - Uses fields such as Name, Competitor Status, Summary, Pricing Summary, Competitor Tracked?, Competitor Sentiment, Mentioned Date, and relation to the sales call page.  
      - Icon set to ü§∫.  
    - Credentials: Notion API "Notion david-internal".  
    - Retry and wait between tries configured.  
  - **Bundle Competitor Data to 1 object**  
    - Type: Aggregate node  
    - Role: Aggregates processed competitor data back into a single object for merging.  
  - **Merge Competitor Thread**  
    - Type: Set node  
    - Role: Merges aggregated competitor data back into the workflow context.  
  - Edge Cases: Empty competitor array, Notion API failures, invalid data fields.

#### 2.5 Objection Data Processing

- **Overview:**  
  Checks for objections in AI output, splits objection tags, formats them for Notion multi-select, updates the call record with objection summary and tags.

- **Nodes Involved:**  
  - Check if objection found  
  - Wait for rate limiting - Objection  
  - Split Out all objections  
  - Format Tag Data for notion  
  - Bundle all Tags  
  - Add Objection Tag to Notion  
  - Wait for rate limiting again  
  - Update Call with Objection Summary  
  - Merge Objection Threads

- **Node Details:**  
  - **Check if objection found**  
    - Type: If node  
    - Role: Checks if the AI output contains a non-empty objection nature string.  
  - **Wait for rate limiting - Objection**  
    - Type: Wait node  
    - Role: Pauses execution for 3 seconds to avoid Notion API rate limits.  
  - **Split Out all objections**  
    - Type: SplitOut node  
    - Role: Splits the objection tags array into individual tags.  
  - **Format Tag Data for notion**  
    - Type: Set node  
    - Role: Formats each objection tag into an object with a `name` property for Notion multi-select compatibility.  
  - **Bundle all Tags**  
    - Type: Aggregate node  
    - Role: Aggregates all formatted tags into a single array.  
  - **Add Objection Tag to Notion**  
    - Type: HTTP Request node  
    - Role: Sends a PATCH request to Notion API to update the multi-select "Objections" property on the call page with objection tags.  
    - Configuration:  
      - Uses Notion API version 2022-06-28.  
      - Authenticated with Notion API credentials.  
  - **Wait for rate limiting again**  
    - Type: Wait node  
    - Role: Pauses 4 seconds to ensure API rate limits are respected before next update.  
  - **Update Call with Objection Summary**  
    - Type: Notion (databasePage update)  
    - Role: Updates the call page's "Objection Summary" rich text field with the objection nature from AI output.  
  - **Merge Objection Threads**  
    - Type: Set node  
    - Role: Merges objection data back into workflow context.  
  - Edge Cases: Empty objection tags, API errors, malformed objection data.

#### 2.7 Rate Limiting and Sequencing

- **Overview:**  
  Implements wait nodes between API calls to Notion to avoid hitting rate limits and ensure smooth sequential processing.

- **Nodes Involved:**  
  - Wait for rate limiting - Competitors  
  - Wait for rate limiting - Integration  
  - Wait for rate limiting - Objection  
  - Wait for rate limiting again  
  - Wait for rate limiting - Use Cases

- **Node Details:**  
  - All are Wait nodes configured with 3 or 4 seconds delay.  
  - Positioned strategically before API-intensive nodes to space out requests.  
  - Edge Cases: If delays are too short, risk of API throttling; if too long, workflow latency increases.

### 4. Reproducing the Workflow from Scratch

1. **Create the Trigger Node**  
   - Add an **Execute Workflow Trigger** node. No parameters needed. This node will receive AI-generated sales call data including `AIoutput` and `notionData`.

2. **Add Core Data Update Node**  
   - Add a **Notion** node configured to update a database page.  
   - Operation: Update page.  
   - Page ID: Set dynamically from `Execute Workflow Trigger` node's `notionData[0].id`.  
   - Properties to update:  
     - Call Summary (rich_text) ‚Üê `AIoutput.CallSummary`  
     - Next Steps (rich_text) ‚Üê formatted bullet list from `AIoutput.NextSteps`  
     - Sentiment (select) ‚Üê `AIoutput.Sentiment`  
     - Customer Pain Points (rich_text) ‚Üê formatted bullet list from `AIoutput.CustomerPainPoints`  
   - Credentials: Connect your Notion API credentials.

3. **Competitor Data Processing**  
   - Add an **If** node named "Check if Competitor Data Found" with condition: length of `AIoutput.Competitors` ‚â• 1.  
   - On true branch:  
     - Add a **Wait** node ("Wait for rate limiting - Competitors") with 3 seconds delay.  
     - Add a **Set** node ("Get AI Response - Competitors") to assign `aiResponse` from `AIoutput`.  
     - Add a **SplitOut** node ("Split Out Competitor Data") to split `aiResponse.Competitors`.  
     - Add a **Notion** node ("Create Competitors in DB") to create pages in the Competitors database:  
       - Database ID: Your Competitors database ID.  
       - Properties: Map fields such as Name, Competitor Status, Summary, Pricing Summary, Competitor Tracked?, Competitor Sentiment, Mentioned Date, and relation to the call page.  
       - Icon: ü§∫  
       - Credentials: Notion API.  
     - Add an **Aggregate** node ("Bundle Competitor Data to 1 object") to aggregate created competitor pages.  
     - Add a **Set** node ("Merge Competitor Thread") to merge aggregated data back.

4. **Integration Data Processing**  
   - Add an **If** node ("Check if Integration Data Found") with condition: length of `AIoutput.Integrations` ‚â• 1.  
   - On true branch:  
     - Add a **Wait** node ("Wait for rate limiting - Integration") with 3 seconds delay.  
     - Add a **SplitOut** node ("Split Out Integration data") to split `AIoutput.Integrations`.  
     - Add a **Notion** node ("Create Integrations") to create pages in the Integrations database:  
       - Database ID: Your Integrations database ID.  
       - Properties: Map fields such as Name, Integration Status, Summary, Usage Status, and relation to the call page.  
       - Icon: ‚öôÔ∏è  
       - Credentials: Notion API.  
     - Add an **Aggregate** node ("Bundle Integration Data to 1 object") to aggregate created integration pages.  
     - Add a **Set** node ("Merge Integration Thread") to merge aggregated data back.

5. **Objection Data Processing**  
   - Add an **If** node ("Check if objection found") with condition: `AIoutput.Objection.Nature` is not empty.  
   - On true branch:  
     - Add a **Wait** node ("Wait for rate limiting - Objection") with 3 seconds delay.  
     - Add a **SplitOut** node ("Split Out all objections") to split `AIoutput.Objection.ObjectionTags`.  
     - Add a **Set** node ("Format Tag Data for notion") to format each tag as `{ "name": "<tag>" }`.  
     - Add an **Aggregate** node ("Bundle all Tags") to aggregate formatted tags.  
     - Add an **HTTP Request** node ("Add Objection Tag to Notion"):  
       - Method: PATCH  
       - URL: `https://api.notion.com/v1/pages/{{pageId}}` where `pageId` is from trigger node.  
       - Body (JSON): Update multi-select property "Objections" with aggregated tags.  
       - Headers: Include `Notion-Version: 2022-06-28`.  
       - Authentication: Use Notion API credentials.  
     - Add a **Wait** node ("Wait for rate limiting again") with 4 seconds delay.  
     - Add a **Notion** node ("Update Call with Objection Summary") to update the call page's "Objection Summary" rich text with `AIoutput.Objection.Nature`.  
     - Add a **Set** node ("Merge Objection Threads") to merge objection data back.

6. **Use Case Data Processing**  
   - Add an **If** node ("Check for use cases") with condition: length of `AIoutput.UseCases` ‚â• 1.  
   - On true branch:  
     - Add a **Wait** node ("Wait for rate limiting - Use Cases") with 3 seconds delay.  
     - Add a **SplitOut** node ("Split Out Use Cases") to split `AIoutput.UseCases`.  
     - Add a **Notion** node ("Create Use Cases") to create pages in the Use Cases database:  
       - Database ID: Your Use Cases database ID.  
       - Properties: Map fields such as Summary (title), Department Tag (multi-select), Industry Tag (multi-select), Implementation (select), and relation to the call page.  
       - Icon: üíº  
       - Credentials: Notion API.  
     - Add an **Aggregate** node ("Bundle Use Case Data to 1 object") to aggregate created use case pages.  
     - Add a **Set** node ("Merge Use Case Thread") to merge aggregated data back.

7. **Connect Nodes**  
   - Connect the trigger node to all four If nodes and the core Notion update node.  
   - Connect each If node's true branch to its respective wait node, then to split, create, aggregate, and merge nodes sequentially.  
   - Connect the objection processing nodes in sequence including the HTTP request and final update.  
   - Ensure all wait nodes are placed before Notion API calls to handle rate limiting.

8. **Credentials Setup**  
   - Configure Notion API credentials in n8n with access to all relevant databases.  
   - Ensure the API key has permissions to read and write pages in Competitors, Integrations, Use Cases, and Calls databases.

9. **Testing**  
   - Test the workflow with sample AI-generated call data containing competitors, objections, integrations, and use cases.  
   - Verify that Notion pages are created/updated correctly and that rate limiting does not cause errors.
